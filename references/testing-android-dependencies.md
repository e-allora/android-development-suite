# Testing Android Dependencies Without Emulator

## The Problem

Android framework classes (`android.speech.tts.TextToSpeech`, `android.graphics.pdf.PdfRenderer`,
`android.net.Uri`, etc.) cannot be instantiated on a plain JVM. Unit tests that
depend on them either need Robolectric, an emulator, or a physical device —
making CI on headless Linux VMs impossible.

## The Pattern: Dependency Inversion Boundary

Extract an **interface** in the domain layer (pure Kotlin, no Android imports)
that abstracts the Android dependency. The production implementation lives in
the data layer and wraps the real Android class. Tests inject a fake.

```
domain/tts/TtsEngine.kt       ← interface (no Android imports)
data/tts/RealTtsEngine.kt     ← wraps android.speech.tts.TextToSpeech
test/.../FakeTtsEngine.kt     ← tracks calls, fires synthetic callbacks
```

### Step 1: Interface in domain/

```kotlin
// domain/tts/TtsEngine.kt — pure Kotlin, testable on JVM
interface TtsEngine {
    data class Voice(val id: String, val locale: String, val label: String)

    interface Listener {
        fun onUtteranceStart(utteranceId: String)
        fun onUtteranceDone(utteranceId: String)
        fun onUtteranceError(utteranceId: String)
    }

    suspend fun init(): Boolean
    fun loadVoices(): List<Voice>
    fun setRate(rate: Float)
    fun setVoice(voiceId: String?)
    fun enqueue(utteranceId: String, text: String)
    fun stop()
    fun shutdown()
    fun addListener(listener: Listener)
    fun removeListener(listener: Listener)
}
```

### Step 2: Production implementation in data/

```kotlin
// data/tts/RealTtsEngine.kt — wraps Android TextToSpeech
@Singleton
class RealTtsEngine @Inject constructor(private val context: Context) : TtsEngine {
    private var engine: TextToSpeech? = null
    private val listeners = mutableListOf<TtsEngine.Listener>()

    override suspend fun init(): Boolean =
        suspendCancellableCoroutine { cont ->
            engine = TextToSpeech(context) { status ->
                val ok = status == TextToSpeech.SUCCESS
                if (ok) engine?.setOnUtteranceProgressListener(bridge)
                if (cont.isActive) cont.resume(ok)
            }
        }
    // ... bridge translates UtteranceProgressListener → TtsEngine.Listener
}
```

### Step 3: Fake for tests

```kotlin
// test/.../FakeTtsEngine.kt — synchronous, tracks calls
class FakeTtsEngine : TtsEngine {
    var initResult = true
    var initCalls = 0
    val enqueueCalls = mutableListOf<Pair<String, String>>()
    val listeners = mutableListOf<TtsEngine.Listener>()

    override suspend fun init(): Boolean { initCalls++; return initResult }
    override fun enqueue(id: String, text: String) { enqueueCalls.add(id to text) }
    // ... etc

    // Test helpers to simulate async callbacks:
    fun fireStart(id: String) = listeners.forEach { it.onUtteranceStart(id) }
    fun fireDone(id: String) = listeners.forEach { it.onUtteranceDone(id) }
    fun fireError(id: String) = listeners.forEach { it.onUtteranceError(id) }
}
```

### Step 4: DI wiring

```kotlin
// TtsModule.kt — production binds real; tests pass the fake directly
@Module @InstallIn(SingletonComponent::class)
abstract class TtsModule {
    @Binds @Singleton
    abstract fun ttsEngine(impl: RealTtsEngine): TtsEngine

    @Binds @Singleton
    abstract fun ttsPlayer(impl: TtsPlayerImpl): TtsPlayer
}
```

## When to Use This Pattern

- Any Android system service that has no JVM equivalent: `TextToSpeech`,
  `PdfRenderer`, `BluetoothAdapter`, `LocationManager`, `CameraManager`
- Repository methods that take `android.net.Uri` — extract to a `FileSource`
  or `InputStream` abstraction
- Any class that constructs Android objects in its constructor (the anti-pattern
  to avoid)

## When NOT to Use

- Room DAOs — these are abstractions already (interfaces generated by Room).
  Test with in-memory Room databases or MockK on the DAO interface directly.
- Simple value classes (`Bundle`, `Intent` extras) — these don't need
  abstraction; just construct them in tests.
- Things that genuinely need Robolectric (Compose UI, resource loading,
  `Application` context) — some things just need the real framework.

## Proven In

- **VoxBook TTS Reader** — `TtsEngine` abstraction made `TtsPlayerImpl` fully
  testable on JVM. 16 unit tests cover the entire state machine without emulator.
- Same pattern applies to `PdfRenderer` (for PDF text extraction) and would
  apply to any future Android system-service integration.
